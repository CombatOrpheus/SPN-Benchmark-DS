## 2024-03-27 - Intermediate Dense Array Bottleneck in State Equation Matrix Construction
**Learning:** Dense intermediate numpy arrays used before constructing sparse arrays (e.g. `scipy.sparse.csc_array(np.zeros(...))`) are major memory and speed bottlenecks when working with graph adjacency matrices, especially when reachability graphs scale as nodes^2 edges. An O(V^2) memory bottleneck can easily crash execution for large benchmark datasets.
**Action:** Always construct sparse arrays directly using format-specific construction arrays like `rows`, `cols`, and `data` in COO format (`scipy.sparse.coo_array((data, (rows, cols)))`) before converting to CSC/CSR. This keeps memory allocation tight (O(E+V)) and minimizes initialization overhead when optimizing large-scale graph equations.
## 2024-05-24 - Numba compute_average_markings Vectorization Bottleneck
**Learning:** Using numpy boolean indexing and `np.unique` inside Numba JIT compiled functions (e.g. `states_with_token = vertices[:, place_idx] == token_val` in loops) creates significant overhead because it instantiates intermediate arrays repeatedly for each state and each unique token.
**Action:** Unroll these loops in Numba by allocating dense arrays and pre-computing mappings (`token_to_idx`), avoiding all `np.unique` and boolean masking inside tight loops. This avoids O(N*P*U) intermediate array creations and resulted in ~2x speedup in marking density computation.
## 2024-05-25 - Numba implicit array allocation and integer multiplication bottlenecks
**Learning:** Using `np.where` and implicit boolean mask creation (e.g., `np.ones(..., dtype=np.bool_)`) inside tight Numba loops during reachability graph BFS causes significant overhead because intermediate arrays are repeatedly allocated and garbage collected. Furthermore, FNV-1a hash functions rely on heavy 64-bit integer multiplications.
**Action:** Always pre-allocate dense arrays and use explicit tracking counters (e.g., `enabled_count`) in hot loops instead of relying on NumPy's high-level boolean indexing or `np.where` when working inside Numba. For simple hashing in inner loops, use DJB2 (`(h << 5) + h`) instead of FNV-1a to avoid 64-bit integer multiplication overhead.
